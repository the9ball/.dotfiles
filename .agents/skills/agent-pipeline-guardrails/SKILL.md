# Agent Pipeline Guardrails

Architect / Implementer / Reviewer / Investigator は個人用サブエージェント(`~/.claude/agents/architect.md` /
`implementer.md` / `reviewer.md` / `investigator.md`)。オーケストレーション専用の自動化スキルは意図的に作っていない
(パイプライン全体を自動連鎖させると、途中の人間の承認ポイントが失われるため)。このスキルは自動化の
代わりに、オーケストレータ(このセッション自身)が守るべき最低限の確認ルールをまとめたもの。

## 守ること

1. **Architect への計画書き換え指示は、ユーザーの明示的な承認を得てから出す。**
   - Implementer/Reviewer/Investigator からのフィードバックを Architect に渡すときは、単に転送するだけでなく、
     フィードバック内容と(あれば)Architect が提案した修正案をユーザーに提示し、承認を得てから
     「この内容で計画を修正して」のように曖昧さのない指示を出す。
   - 「対応して」「良きに計らって」のような曖昧な依頼で済ませない。Architect 自身も「明示指示が
     無ければ書き換えない」という基準で動くよう定義されているので、オーケストレータ側が先に承認を
     飛ばしても大事故にはなりにくいが(Architect 側が提案止まりにする)、ユーザーの意図を確認せずに
     話を進めてしまうことに変わりはない。

2. **Implementer の実装を本番採用として扱う前に、ユーザーの明示判断を得る。**
   - プロダクト側の制約(AI 生成コードは本番に含められない。デバッグ用の一時コードのみ、都度の判断で
     許可される)により、Implementer の最終報告(本番/デバッグ分類・変更ファイル一覧)は必ずユーザーに
     提示し、採否を確認してから次に進む。
   - 却下する場合は、Implementer が報告した「却下時の戻し方」(既存ファイルは `git checkout`/`restore`、
     新規ファイルは削除)に従う。

3. **迷ったら止める。**
   - 何が承認されたのか自分で不確かな場合、確認を省略せず、ユーザーに一度尋ね直す。

4. **フィードバックを鵜呑みにせず、中継前に自分で不明点を確認する。**
   - Reviewer/Implementer/Investigator からのフィードバックを別のエージェント(特に Architect)に中継する前に、内容を
     読んで自分が本当に理解できているか確認する。曖昧・矛盾している・根拠が薄いと感じた箇所を、都合よく
     解釈・要約して埋めない。
   - 不明点があれば、先に(Architect などへ中継する前に)ユーザーに確認する。ユーザーへの確認を省略して
     推測で中継すると、その推測がそのまま計画やコードに反映されてしまう。

## いつ Codex(Implementer)をデフォルトにするか

判断基準は単純な二値: **このスレッド内で、このスキル(agent-pipeline-guardrails)が一度でも読み込まれたか**。
これは「このスレッドで既に Architect/Implementer/Reviewer のいずれかを使い、計画ファイルを作成・参照
済みである」ことの目印として扱ってよい。

- **このスレッドで一度でも読み込まれていれば**: それ以降このスレッド内での実装(コード変更)作業は、
  単純な1行修正など明らかに軽微なものを除き、自分で Edit/Write するのではなく、既定で Implementer
  経由で Codex に委譲する。「Implementerで」という明示的な指名は不要。計画がまだ無い新規タスクが
  スレッド内で出てきた場合は、まず Architect で計画を立てて承認を得てから Implementer に進む(この
  順序は省略しない)。
- **このスレッドで一度も読み込まれていなければ**: この既定ルーティングは適用しない。Codex/Implementer
  のことは一切気にせず、これまで通り自分で直接コードを書いてよい。

## Codex に渡すモデルの指定

`implementer.md` 自体にはモデル名をハードコードしない(モデルは変更・廃止されやすく、ファイルが陳腐化する
ため)。代わりに、このスレッド内でユーザーが Codex 委譲時に使うモデルを明示的に指定したら、それを以降
このスレッド内の Implementer への委譲(`codex:codex-rescue` に渡すタスク文の `--model <name>`)に反映する。
スレッドを跨いでは持ち越さない(新しいスレッドではまた指定が無ければ既定(モデル指定なし)に戻る)。

## PR 作成前のデュアルレビュー(Claude + Codex の総合)

「PR 前の最終レビュー」「Claude と Codex 両方でレビューして」といった依頼のときは、オーケストレータ(このセッション自身)が Claude 側と Codex 側の2つのレビューを走らせ、結果を総合して1つのレビューとして返す。合成は両レビューの上に立つオーケストレータの役割であり、どちらか一方のエージェントに合成までやらせない。

- **Claude 側**: `reviewer` エージェントを通常どおり `Agent`(`run_in_background: false`)で呼ぶ。Reviewer 定義は変更しない(単独・独立のレビューという役割を保つ)。
- **Codex 側**: `codex:codex-rescue` を `Agent`(`run_in_background: false`)で呼び、**read-only のレビュー**を依頼する(タスク文に `--write` を付けない、コードを変更させない旨を明記)。
  - 注意: ネイティブの `/codex:review` / `/codex:adversarial-review` は `disable-model-invocation` のためオーケストレータからは起動できない。自動で回せるのは `codex:codex-rescue` 経由のレビューのみ。ネイティブの構造化レビューが欲しい場合は、その旨をユーザーに伝え、ユーザー自身にスラッシュコマンドを実行してもらう(結果を貼ってもらえれば合成に含める)。
- **並行実行**: 2つの `Agent` 呼び出しは(依存が無いので)できるだけ同じターンで並行に投げ、両方の結果が揃ってから合成する。
- **合成して返す**: 両者の指摘を突き合わせ、(a) 両方が挙げた指摘(確度高)、(b) 片方だけが挙げた指摘(出所を明示)、(c) 両者で判断が食い違う点、を整理して1つの総合レビューにする。安易に片方を正としない。食い違いは「どちらの言い分か」を残したまま提示し、最終判断はユーザーに委ねる。
- **最終レビューでも修正はしない**: これはあくまでレビュー。指摘に基づく修正は別途 Implementer 経由で行い、本番採用の可否はユーザーが判断する(このスキルの他のガードレールと同じ)。

## なぜ

このガードレールはプロンプトの表現(何をもって「承認」とするか)を明確化するだけであり、技術的な
強制力はない。Architect 自身も過去に「対応してください」という曖昧な依頼を書き換えの許可と誤って
解釈したことがあり(2026-07-10 のテストで発見・修正済み)、オーケストレータ側にも同じ種類のリスクが
ある。技術的に権限を制限できる箇所(コード編集ツールの有無、作業ディレクトリの範囲、既定モデルなど)は
既に各エージェント定義側の `tools`/本文ルールで対応済み。ここでカバーしているのは、それでは塞ぎきれない
「言葉の意味の解釈」に関わる部分だけ。
